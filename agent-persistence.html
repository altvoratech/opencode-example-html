Onde os agentes guardam memória

Todo agente de IA de terminal persiste sessões no disco. Persistência não é diferencial de ninguém — todos guardam. O que muda, e o que realmente importa para construir recall em cima, é o shape: a forma como cada um organiza esses dados. Existem duas filosofias, e este documento mostra as duas com dados reais lidos do disco.

Como esses números foram obtidos

Tudo aqui veio de inspecionar diretamente o sistema de arquivos (find, ls, parsing de JSON) — não de documentação. A verdade está no disco, não no docs.

A surpresa

O Claude Code, que parecia ser o que não tinha persistência, é na verdade o campeão de acúmulo: 691 sessões registradas, a maior com 12 MB. O que parecia não guardar é o que mais guarda.

As duas filosofias

A diferença real não é "tem ou não tem" — é como. Toda a tabela acima colapsa em dois modelos opostos de organizar a mesma informação.

Event log append-only

Uma sessão = um arquivo .jsonl = um evento por linha. Você só acrescenta ao fim. Tudo (mensagem, anexo, snapshot, permissão) vive intercalado no mesmo fluxo. Para reconstruir o estado, você reproduz a stream.

Claude Code · Codex · Hermes

Document store normalizado

Entidades separadas em arquivos próprios (session, message, part), em diretórios hierárquicos, joináveis por ID. Acesso aleatório e quase pronto para query.

Filosofia 1 · O stream (Claude Code)

Uma única sessão do Claude Code é um arquivo .jsonl. Cada linha é um evento autocontido. Na maior sessão deste disco, os tipos de evento se distribuem assim — repare como mensagem, anexo, snapshot de arquivo e mudança de permissão dividem o mesmo fluxo:

Snapshot dentro do log

Note os 399 file-history-snapshot: o Claude Code versiona o estado dos arquivos dentro da própria stream. É o equivalente ao diretório snapshot/ do OpenCode — mesma ideia, lugar diferente.

Filosofia 2 · A árvore (OpenCode)

O OpenCode quebra a mesma informação em entidades separadas, aninhadas por ID. A relação é project → session → message → part, com chaves estrangeiras embutidas:

part/msg_<messageID>/prt_*.json · 4.745

Metadado de graça

O store já entrega, sem custo, o que um recall normalmente teria que calcular: modelo e agente por mensagem (model{providerID, modelID}), e volume de edição por sessão (summary{additions, deletions, files}). Tem ainda um marcador migration + campo version — ou seja, o formato migra entre versões, e quem lê precisa ser version-aware.

Stream vs. árvore — o trade-off

Nenhum é "melhor". Cada filosofia otimiza um lado e paga no outro:

CritérioEvent logDocument store
Custo de escritaMínimo — só dá appendMaior — vários arquivos por turno
Consulta pontualCara — parsear a stream todaBarata — acesso direto por ID

Acesso aleatórioFracoForte

O que isso significa para um recall

Se a meta é um recall universal — que leia memória de qualquer agente — a consequência é direta: ele precisa falar as duas línguas.

Para os streams

Claude Code, Codex, Hermes → um passe de parse linha a linha, agrupando por sessionId e indexando antes de consultar.

Para o store

OpenCode → walk dos diretórios session → message → part, já joinável. Metade do trabalho de um recall já está escrita pela própria ferramenta.

O detalhe do projectID

Cuidado com a identidade de projeto: o OpenCode chaveia por um hash (445c16…, o mesmo dos snapshots), enquanto recalls costumam chavear por git URL / UUID próprio. Alinhar os dois exige um mapeamento hash ↔ git-url no meio.

Conclusão

Persistência não diferencia ninguém — todos guardam. O diferencial é o shape, e é ele que decide o quão fácil é construir recall em cima. Antes de escrever um único byte de storage novo, vale olhar o que o agente já deixou no disco.

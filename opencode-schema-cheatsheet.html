1. Como ler um JSON Schema

O JSON Schema é um contrato declarativo que descreve a forma de um JSON. O opencode usa ele para validar opencode.json na inicialização — se algo viola o schema, o programa nem sobe.

O schema é um JSON normal com keywords especiais. As principais:

KeywordO que fazAnalogia TS
typeDeclara o tipo primitivostring / number / boolean / object / array
propertiesCampos de um objetointerface { ... }
requiredCampos obrigatoriosCampo sem ? na interface
additionalPropertiesSe false: objeto fechado. Se {"$ref"}: chaves soltas seguem outra definição{ [key: string]: T } vs interface fechada
enumValores literais permitidostype X = "a" | "b" | "c"
anyOfUnião permissiva — pelo menos um matchstring | number
oneOfUnião exclusiva — exatamente um matchSem equivalente direto; mais restrito que |
allOfComposição — todos devem passar (intersecção)A & B
$refReferencia a outra definiçãoSomeType (import)
$defsBloco de definições reutilizaveisBloco type X = ... no topo
Leitura pratica: Quando voce ve "$ref": "#/$defs/AgentConfig", leia como AgentConfig. Quando ve "anyOf": [{"type":"string"}, {"type":"boolean"}], leia como string | boolean. O schema é uma AST, nao um formato de dados — ele descreve o formato valido.

O prefixo $ em $schema, $ref, $defs é convenção do JSON Schema. Nao tem nada a ver com o loop de runtime nem com variaveis de template.

2. Equivalencia mental com TypeScript

Se voce pensa em TypeScript, aqui esta a tradução direta de cada padrao do schema do opencode:

Schema (JSON)TypeScript equivalente
"type": "string", "enum": ["a","b"] "a" | "b"
{ "type": "object", "properties": { "x": {...} }, "additionalProperties": false } interface { x: ... } (fechado)
{ "type": "object", "additionalProperties": { "$ref": "#/$defs/X" } } { [key: string]: X }
"anyOf": [{ "type": "boolean" }, { "type": "string", "enum": ["notify"] }] boolean | "notify"
"allOf": [{ "$ref": "#/$defs/A" }, { "$ref": "#/$defs/B" }] A & B
{ "$ref": "#/$defs/AgentConfig" } AgentConfig (import/referencia)
"required": ["a", "b"] Campos sem ?
description no schema Comentario JSDoc (/** ... */)
"$defs": { "X": {...}, "Y": {...} } type X = ...; type Y = ...; (declarações no modulo)
Cuidado: Ao contrario do TypeScript, JSON Schema nao nomeia tipos automaticamente. $defs é necessario para reuso. E anyOf é mais permissivo que oneOf — oneOf falha se mais de uma alternativa casar.

3. Config / Schema / Runtime

config instance → schema validation → runtime behavior

Sao tres camadas distintas que e facil confundir:

CamadaO que éExemplo
Config O JSON que voce escreve em opencode.json "default_agent": "raphael"
Schema O contrato em https://opencode.ai/config.json que valida a config "default_agent": { "type": "string", "description": "..." }
Runtime O motor que executa o comportamento baseado na config Ler default_agent e redirecionar o chat
Regra pratica: O schema descreve forma (o que pode ser escrito). O runtime executa comportamento (o que acontece quando é lido). Um campo pode ser valido no schema mas ignorado no runtime (ex: autoshare — deprecated, schema ainda aceita, runtime nao usa).

O schema do opencode tem uma raiz ($defs.Config) que e o top-level opencode.json. Dentro dela, additionalProperties: false — chaves desconhecidas sao rejeitadas com ConfigInvalidError.

4. AgentConfig explicado

Conheca $defs.AgentConfig — o tipo mais importante do schema. Ele descreve cada entrada dentro de agent: { ... }.

{
  "model"?      : string,      // "provider/model-id"
  "variant"?    : string,      // variante default do modelo
  "temperature"? : number,
  "top_p"?      : number,
  "prompt"?     : string,      // system prompt inline
  "description"?: string,      // quando usar o agente
  "mode"?       : "subagent" | "primary" | "all",   // PAPEL
  "hidden"?     : boolean,     // oculta do @-menu
  "disable"?    : boolean,     // desliga built-in
  "color"?      : string,      // "#FF5733" | "primary" | "error"
  "steps"?      : number,      // max iteracoes antes de fallback
  "permission"? : PermissionConfig,
  "options"?    : object       // extra, roteado sem validação
}
mode aqui e o AgentConfig.mode:
  • "subagent" — so pode ser chamado por outro agente, nao aparece como entrada de chat
  • "primary" — entrada propria no seletor de agente, pode iniciar conversas
  • "all" — ambos: entrada propria e chamaavel por outros agentes

Como definir um agente

Inline (dentro do opencode.json):

"agent": {
  "my-agent": {
    "mode": "subagent",
    "model": "anthropic/claude-sonnet-4-6",
    "description": "Faz X",
    "prompt": "Voce e um especialista em X."
  }
}

Arquivo (recomendado para prompts longos):

.opencode/agents/my-agent.md  ou  .opencode/agent/my-agent.md

---
description: Faz X
mode: subagent
model: anthropic/claude-sonnet-4-6
---

Voce e um especialista em X. (body vira o prompt)

Agentes built-in

O schema lista plan, build, general, explore, scout (experimental), mais os internos title, summary, compaction. Para desabilitar um built-in:

"agent": {
  "build": { "disable": true }
}

5. agent vs mode (legado) vs AgentConfig.mode

Esse e o ponto que mais confunde. Tres coisas diferentes com o mesmo nome:

O queOnde viveO que fazStatus
agent Top-level: "agent": { ... } Dicionario de todos os agentes configurados (built-in + custom) atual
mode (top-level) "mode": { "plan": AgentConfig, "build": AgentConfig } Mesmo que agent mas legado/deprecated. Usava mode.plan e mode.build antes de existir agent. Ainda aceito, mas prefira agent. deprecated
AgentConfig.mode Dentro de agent: { nome: { mode: ... } } Enum "subagent" | "primary" | "all" — papel do agente ativo
Cuidado! O mode top-level (objeto com plan e build) e diferente do mode dentro de AgentConfig (string literal). Eles convivem no schema mas significam coisas opostas. Um e container, o outro e "papel do agente". 
// ATUAL — use agent
"agent": {
  "my-agent": { "mode": "subagent", ... }
}

// LEGADO — mode top-level ainda funciona mas e deprecated
"mode": {
  "plan": { ... }
}

// O schema aceita ambos; o runtime mergeia.

Plugins

Plugin estende o motor com hooks — nao cria tools novas (a nao ser que o hook tool seja usado). A config de plugin e uma array:

Input / Output / Prompt

Conceitos que aparecem em agentes, plugins e commands:

ConceitoO que é
PromptInstrução do sistema dada ao modelo. AgentConfig.prompt ou frontmatter prompt: (ou body do .md)

OutputDado que sai de uma operação. Mutação output.args altera o que a tool recebe

Em plugins, o padrao e: receba input, leia/mute output, retorne void. Nao retorne valores — mutação in-place.

7. Partes do schema: plugin, permission, mcp, provider, command

plugin array

Array de strings ou tuplas [name, options]. Cada item e um anyOf entre string e tupla. O schema nao valida o conteudo do arquivo — so a forma da declaração.

FormatoSchema equivalente
"plugin": ["pkg-name"]anyOf: [string, [string, object]]
"plugin": [["pkg", {opt: 1}]]Tupla com prefixItems de [string, object]

permission $ref: PermissionConfig

O shape mais aninhado do schema. PermissionConfig e um anyOf entre:

  • ActionConfig — string "allow" | "ask" | "deny" (permissao global unica)
  • Object config — objeto com chaves por tool, cada valor sendo PermissionRuleConfig

PermissionRuleConfig por sua vez e anyOf entre:

  • "allow" | "ask" | "deny" (shorthand para {"*": value})
  • PermissionObjectConfig — objeto de pattern → action, ultima regra vence
// Formas validas:
"permission": "allow"                        // tudo allow (raro, agressivo)
"permission": { "edit": "deny" }             // nega edit, resto default
"permission": {
  "bash": {
    "*": "ask",                              // catch-all PRIMEIRO
    "git *": "allow",                        // especifica depois
    "rm *": "deny"
  }
}
Ordem importa! Em PermissionObjectConfig, a ultima regra que match vence. Coloque o catch-all * primeiro e as regras especificas depois.

mcp $ref: McpLocalConfig | McpRemoteConfig

Objeto chaveado por nome de servidor MCP. Discriminado por type:

"mcp": {
  "my-local": {
    "type": "local",
    "command": ["npx", "-y", "@playwright/mcp"],
    "enabled": true,
    "environment": { "KEY": "val" }           // NOTA: "environment", nao "env"
  },
  "my-remote": {
    "type": "remote",
    "url": "https://...",
    "headers": { "Authorization": "Bearer ..." },
    "oauth": { "clientId": "...", "scope": "..." }  // ou false para desligar
  }
}

Cada valor do objeto mcp na verdade e um anyOf entre:

  • Servidor real (McpLocalConfig | McpRemoteConfig)
  • Objeto { enabled: boolean } — util para desabilitar servidor herdado
MCP usa environment, nao env
O schema de McpLocalConfig chama o campo de variaveis de ambiente de environment. command e sempre um array de strings, nunca uma string unica. type e obrigatorio para servidores reais.

provider $ref: ProviderConfig

Objeto chaveado por nome do provider. Cada provider pode ter options (API keys, baseUrl, etc.), models para overrides, e headers customizados.

"provider": {
  "anthropic": {
    "options": { "apiKey": "sk-ant-..." }
  }
}

Ha tambem disabled_providers (desliga providers carregados automaticamente) e enabled_providers (whitelist — so estes rodam).

command object keyed

Commands customizaveis (ex: /deploy). Objeto onde cada chave e um comando, com template obrigatorio:

"command": {
  "deploy": {
    "template": "Faca deploy do {branch} no ambiente de staging",
    "description": "Gera prompt de deploy",
    "agent": "build",           // opcional: delegar para agente especifico
    "model": "...",             // opcional: modelo especifico
    "subtask": true             // opcional: roda como sub-tarefa
  }
}

8. Tutorial pratico: criando o MyPlugin

Este tutorial reconstitui o processo real de criação e depuração do plugin plugins/my-plugin.ts. O objetivo era simples: prefixar todo comando bash com um echo visível para confirmar que o plugin está ativo durante a execução. O que parecia trivial revelou armadilhas conceituais do sistema de hooks.

Objetivo

Injetar echo "[MyPlugin] plugin ativo" && antes de cada comando bash executado pelo opencode. O usuário vê a mensagem no terminal e sabe que o hook funcionou.

Primeira abordagem (errada)

A tentativa inicial foi interceptar a tool read:

import type { Plugin } from "@opencode-ai/plugin";

export const MyPlugin: Plugin = async (_ctx) => {
  return {
    "tool.execute.before": async (input, output) => {
      if (input.tool === "read") {
        // Tentativa errada: read nao usa "command"
        output.args.command = "echo ativo && ${original}";
      }
    },
  };
};
Não funcionou — eis por quê:
  • A tool read não usa o campo command. read espera path ou filePath — não existe args.command nela.
  • O campo extra command foi simplesmente ignorado pelo runtime. O motor só lê os campos que cada tool reconhece; qualquer campo extra é descartado sem erro.
  • Nenhum comando foi alterado, nenhum echo apareceu — e sem mensagem de erro. O hook executou, mas a mutação não surtiu efeito.
Lição 1: Cada tool tem seu próprio conjunto de argumentos. Mutar output.args só funciona se o campo alterado for realmente lido pelo runtime daquela tool. Conheça os args de cada tool antes de interceptar.

Escolha da tool correta

O comando que queremos prefixar é executado pelo shell — logo, a tool certa é bash. É a única tool do opencode que usa args.command como campo de execução.

A correção foi trocar a guarda de input.tool === "read" para input.tool === "bash".

Estrutura do hook tool.execute.before

O hook recebe dois parâmetros:

ParâmetroO que contémUso no MyPlugin
inputMetadados da chamada: input.tool (nome da tool), input.args (argumentos originais, apenas-leitura)Filtrar qual tool interceptar via input.tool
outputObjeto output.args que pode ser mutado para alterar os argumentos que a tool veráInjetar o prefixo em output.args.command
Dica: input é apenas-leitura por convenção. output é o objeto de mutação. Não retorne valores — a comunicação é feita por mutação in-place de output.args.

A guarda de tipo

Mesmo com a tool certa, output.args.command pode ser undefined (chamadas internas podem não ter command como string simples). Sem a guarda, o código quebraria com TypeError.

// Correto: guarda de tipo antes de mutar
if (typeof output.args.command === "string") {
  const original = output.args.command;
  output.args.command = `echo "[MyPlugin] plugin ativo" && ${original}`;
}
Pegadinha: Sem o typeof check, o TypeScript até compila (se o tipo de command for string | undefined), mas em runtime undefined quebra a interpolação. A guarda não é opcional — é obrigatória para plugins robustos.

Código final

Versão completa e funcional do plugin:

import type { Plugin } from "@opencode-ai/plugin";

// Plugin mínimo: prefixar todo comando bash com um echo visível,
// para confirmar que o hook está ativo durante a execução.
export const MyPlugin: Plugin = async (_ctx) => {
  return {
    "tool.execute.before": async (input, output) => {
      // Só intercepta a tool "bash" — outras tools passam sem alteração
      if (input.tool === "bash") {
        // Guarda de tipo: só altera se command for string (evita undefined)
        if (typeof output.args.command === "string") {
          const original = output.args.command;

          // Prefixa o comando original com um echo identificador,
          // assim o usuário vê "[MyPlugin] plugin ativo" no terminal
          output.args.command = `echo "[MyPlugin] plugin ativo" && ${original}`;
        }
      }
    },
  };
};

Ativação no opencode.json

Registre o plugin no opencode.json:

"plugin": [
  "./plugins/my-plugin.ts"
]
Plugins exigem restart: Diferente de algumas configs que hot-reload, plugins são carregados uma única vez na inicialização. Qualquer alteração no .ts ou no opencode.json requer restart completo do opencode para surtir efeito. Não adianta salvar o arquivo — é necessário reiniciar o processo.

Lições práticas

Checklist mental para criar hooks tool.execute.before:
  1. Identifique qual tool executa a operação que você quer interceptar (use a tabela de tools nativas como referência).
  2. Descubra quais argumentos essa tool aceita — mutar um campo que ela não lê é inútil.
  3. Filtre com input.tool === "nome" para não afetar outras tools.
  4. Adicione guarda de tipo em cada campo de output.args antes de mutar.
  5. Reinicie o opencode para carregar o plugin atualizado.
  6. Teste com um comando simples (ex: echo test) para confirmar que o hook disparou.
Debugging: Se o plugin não funciona, o diagnóstico é simples: (1) verifique se o arquivo existe no caminho declarado, (2) confira se opencode.json é válido (sem erros de vírgula ou chave), (3) reinicie o opencode, (4) execute echo test no terminal — se a mensagem do plugin não aparecer, o hook não está sendo chamado. Não há log interno de plugin — a confirmação é a própria alteração no comportamento.

9. Pegadinhas / Gotchas

Top-level mode ≠ AgentConfig.mode
O schema tem dois campos chamados mode. Um e objeto (mode: { plan: ..., build: ... }) — deprecated. O outro e enum "subagent"|"primary"|"all" dentro de AgentConfig. O nome igual e coincidencia infeliz.
MCP usa environment, nao env
O schema de McpLocalConfig chama o campo de variaveis de ambiente de environment. Copiar de exemplo errado quebra o startup.
plugin e array, nao objeto
Diferente de agent, mcp e provider que sao objetos chaveados, plugin e array de strings ou tuplas. Nao da para dar nome nem desabilitar um plugin individual sem remove-lo.
Permissao: ultima regra vence, nao primeira
Diferente do que a intuição sugere, a ultima regra que match define a permissao. Coloque o catch-all "*" primeiro e as regras especificas depois.
AgentConfig.options aceita qualquer coisa
O schema define "options": { "type": "object" } sem additionalProperties: false. Qualquer chave extra desconhecida no frontmatter de um agente .md e roteada para options.
tools dentro de AgentConfig e deprecated
O schema tem "tools": { "type": "object", "additionalProperties": { "type": "boolean" } } mas a descrição diz @deprecated Use 'permission' field instead.
additionalProperties: false na raiz
O Config top-level tem additionalProperties: false. Qualquer chave nao listada causa ConfigInvalidError. Typos como "defalt_agent" (vs "default_agent") quebram o startup.
model sempre com provider prefix
O formato esperado na pratica e "provider/model-id". Ex: "anthropic/claude-sonnet-4-6". Sem o prefixo, o runtime nao sabe qual provider usar.
Config nao hot-reload
Mudancas no opencode.json so valem depois de reiniciar o opencode. A sessao ativa mantem a config que foi carregada no startup.

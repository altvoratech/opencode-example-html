Runtime do Opencode

Ciclo de vida de uma requisição: prompt → agente → tools → resposta. Documentação do motor de execução, roteamento, plugin hooks e fluxo de tool calling.

Visão Geral

O Opencode é um runtime para agentes de IA que orquestra o ciclo completo de uma requisição: desde o prompt do usuário até a execução de ferramentas e o retorno da resposta. O sistema é dividido em três camadas principais:

Config

JSON que o usuário escreve (opencode.json, frontmatter .md)

Schema

Contrato de validação — JSON Schema em opencode.ai/config.json

Runtime

Motor que executa o comportamento baseado na config + schema

Fluxo de Execução

O runtime executa em 4 etapas principais. Cada etapa é detalhada abaixo com seus componentes e responsabilidades.

1 Usuário · Prompt entrada

O usuário envia uma mensagem de texto para o sistema. A mensagem pode incluir um @-mention para direcionar a um agente específico. O prompt bruto é recebido pelo router, que decide qual configuração de agente será usada.

Formato típico: @agent-nome faça X ou mensagem direta para o agente padrão.
▾
2 Roteamento router

O Router lê a configuração default_agent ou o @-mention para selecionar o AgentConfig e o system prompt correspondente. É aqui que o sistema decide qual agente executará a requisição.

CampoDescrição
default_agentAgente usado quando nenhum @-mention é fornecido
agentsMapa de nomes para configurações de agente
system_promptPrompt de sistema específico do agente selecionado
▾

3. Agent Loop (LLM + Tool Calling)

O coração do runtime. O agente entra em um loop iterativo que processa a requisição até obter uma resposta final ou atingir o limite de steps. Cada iteração compreende:

a LLM · Think

Modelo processa contexto + prompt e decide o próximo passo.

b Decide · Tool Call?

Modelo escolhe tool e argumentos, ou produz resposta final.

c Plugin Hooks

tool.execute.before e tool.execute.after são disparados.

d Tool Execution

read · edit · bash · glob · grep — com permissão allow/ask/deny.

Limite de Iterações

O agent loop possui um limite máximo de steps configurável. Quando atingido, a resposta parcial é retornada com um aviso. Isso evita loops infinitos em casos de tool calling mal-sucedido.

▾
4 Output · Resposta saída

A resposta do modelo é combinada com os resultados das tools executadas e exibida no chat para o usuário. O ciclo se encerra aqui — ou recomeça com um novo prompt.

Plugin Hooks

O sistema de plugins do Opencode permite interceptar a execução de tools através de hooks. Cada plugin pode registrar callbacks para antes e depois da execução de qualquer tool.

HookMomentoInputMutável?
tool.execute.beforeAntes da tool rodar{ tool, args }❌ Somente-leitura
tool.execute.afterDepois da tool rodar{ tool, args, result }✅ args mutável in-place
Exemplo: MyPlugin

O hook tool.execute.before pode injetar "echo [MyPlugin] plugin ativo &&" em output.args.command do bash, permitindo log automatizado de toda execução.

TypeScript 
// Registrar hook antes da execução da tool
runtime.on("tool.execute.before", (ctx) => {
  console.log(`[Plugin] Tool ${ctx.tool} será executada`);
  if (ctx.tool === "bash") {
    ctx.args.command = `echo [MyPlugin] plugin ativo && ${ctx.args.command}`;
  }
});

// Registrar hook depois da execução
runtime.on("tool.execute.after", (ctx) => {
  console.log(`[Plugin] Tool ${ctx.tool} executada em ${ctx.duration}ms`);
});

Configuração

O runtime é configurado através do arquivo opencode.json na raiz do projeto, ou via frontmatter YAML em arquivos .md.

JSON 
{
  "default_agent": "assistente",
  "agents": {
    "assistente": {
      "system_prompt": "Você é um assistente útil.",
      "tools": ["read", "edit", "bash", "glob", "grep"],
      "max_steps": 25,
      "permission": "ask"
    },
    "dev": {
      "system_prompt": "Você é um engenheiro de software.",
      "tools": ["read", "edit", "bash", "glob", "grep"],
      "max_steps": 50,
      "permission": "allow"
    }
  },
  "plugins": [
    {
      "name": "my-plugin",
      "hooks": ["tool.execute.before", "tool.execute.after"]
    }
  ]
}

Campos da Configuração

CampoTipoDescrição
default_agentstringNome do agente padrão
agentsRecord<string, AgentConfig>Mapa de configurações de agente
agents.*.toolsstring[]Lista de tools disponíveis
agents.*.max_stepsnumberLimite máximo de iterações do loop
agents.*.permission"allow" | "ask" | "deny"Política de permissão para execução de tools
pluginsPluginConfig[]Lista de plugins registrados

Exemplos de Uso

Requisição Simples

Enviar um prompt para o agente padrão:

Bash 
$ opencode "Explique o conceito de runtime no contexto de agentes de IA"

Requisição com Agente Específico

Usar @-mention para selecionar um agente:

Bash 
$ opencode "@dev refatore este módulo para usar async/await"

Plugin Customizado

Registrar um plugin que monitora execuções:

JavaScript 
// opencode.config.js
module.exports = {
  plugins: [
    {
      name: "logger",
      hooks: {
        "tool.execute.before": (ctx) => {
          console.log(`[${new Date().toISOString()}] ${ctx.tool} iniciou`);
        },
        "tool.execute.after": (ctx) => {
          console.log(`[${new Date().toISOString()}] ${ctx.tool} finalizou`);
        }
      }
    }
  ]
};

Glossário do Runtime

Termos e definições essenciais para entender a arquitetura do Opencode.

Config
Arquivo JSON que o usuário escreve para definir agentes, tools, plugins e permissões. Ex: opencode.json, frontmatter .md.
Schema
Contrato de validação (JSON Schema) que define a estrutura esperada do arquivo de configuração. Publicado em https://opencode.ai/config.json.
Runtime
Motor que executa o comportamento do sistema baseado na configuração + schema. Gerencia o ciclo de vida da requisição, o agent loop, os hooks e a execução de tools.
Agent Loop
Loop iterativo onde o modelo LLM processa contexto, decide se chama uma tool ou produz resposta final, e repete até atingir um dos critérios de parada.
Plugin Hooks
Pontos de extensão que permitem interceptar e modificar o comportamento do runtime antes e depois da execução de tools.
Tool Calling
Mecanismo pelo qual o modelo LLM solicita a execução de uma ferramenta específica (leitura de arquivo, edição, execução de comando, etc.) e aguarda o resultado para continuar o processamento.
Permission
Sistema de controle que define se uma tool pode ser executada automaticamente (allow), precisa de confirmação (ask) ou é bloqueada (deny).
